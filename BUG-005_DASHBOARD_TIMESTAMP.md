# BUG-005: Dashboard Date Staleness — Root Cause Analysis

## Root Cause

**Primary cause: `datetime.datetime.now()` returns UTC on GitHub Actions runners, but the displayed timestamp implies WIB (UTC+7).**

The dashboard header timestamp is generated by `datetime.datetime.now()` in `scripts/generate_dashboard_v2.py:106`:

```python
date_short = datetime.datetime.now().strftime('%Y-%m-%d %H:%M')
```

On the `ubuntu-latest` GitHub Actions runner, Python's `datetime.datetime.now()` returns **UTC time** (the container has no local timezone set). This UTC timestamp is baked into the static HTML at generation time, and users viewing from WIB (UTC+7) see a time that appears ~7 hours stale.

The dashboard timestamp **exactly matches** the commit timestamp in UTC. This confirms `datetime.datetime.now()` is returning UTC on the runner.

### Secondary issue: No `report_date` fallback

The `report_date` field exists in `turnaround_summary.json` (`"date": "2026-06-07"`) and is available as `summary_data.get('date', 'N/A')` at line 348, but it is **never used** for the header timestamp. The header uses `datetime.datetime.now()` instead.

### Tertiary issue: `file_age()` also uses naive `datetime.datetime.now()`

The Pipeline Diagnostics section shows "0d 0h ago" for file ages (lines 344-346) because `file_age()` at line 98-103 also uses `datetime.datetime.now()`. When the local system and file modification times are close, the age reads zero. On GH Actions, this function also uses UTC, making it inconsistent.

## Fix Recommendation

### Option A (Recommended): Use `report_date` from data instead of `datetime.datetime.now()`

```python
# Line 106 — replace datetime.now() with report_date from data
date_short = summary.get('date', datetime.datetime.now().strftime('%Y-%m-%d'))
date_short += f" {datetime.datetime.now().strftime('%H:%M')}"
```

This ensures the date shown matches the actual data date. The time portion could be dropped or set to a fixed marker (e.g., "after market close").

### Option B: Explicit WIB timezone

```python
import pytz  # or zoneinfo in Python 3.9+
wib = datetime.timezone(datetime.timedelta(hours=7))
date_short = datetime.datetime.now(wib).strftime('%Y-%m-%d %H:%M')
```

This fixes the display by making `now()` return WIB time regardless of runner locale. Requires adding `pytz` dependency or using `zoneinfo` (stdlib 3.9+).

### Option C: Record pipeline run timestamp in JSON

Save an explicit `generated_at` field (in WIB) when generating `turnaround_summary.json`, then read it in `generate_dashboard_v2.py` instead of calling `datetime.datetime.now()`.

This decouples display time from generation environment and is the most robust fix.

### Pipeline Diagnostics fix

```python
# Replace datetime.datetime.now() with UTC-aware comparison
mtime = datetime.datetime.fromtimestamp(path.stat().st_mtime, tz=datetime.timezone.utc)
age = datetime.datetime.now(datetime.timezone.utc) - mtime
```

Or store the pipeline run timestamp once and use it consistently across all time-dependent calculations.
